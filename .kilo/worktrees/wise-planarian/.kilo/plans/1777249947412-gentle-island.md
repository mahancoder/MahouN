# NLP Evidence Extraction and Knowledge Graph Construction Audit Plan

## Objective
Perform a rigorous technical audit of the NLP Evidence Extraction and Knowledge Graph Construction layer of the legal AI system, focusing on:
- evidence extraction reliability
- graph integrity
- test coverage
- pipeline robustness
- failure modes
- hardening against incorrect or corrupted outputs

## Audit Sections

### SECTION 1 — NLP Pipeline Reconstruction
- Reverse-engineer the full evidence extraction pipeline.
- Identify stages: document parsing, preprocessing, text normalization, semantic segmentation/chunking, NER, coreference resolution, relation extraction, event/obligation extraction, entity normalization, evidence attribution, graph entity preparation.
- For each stage: identify implementing module, output data structures, next consumer.

### SECTION 2 — Evidence Integrity and Provenance
- Evaluate traceable legal evidence: each extracted entity/relation should preserve document_id, clause/article_id, paragraph index, character span, original text snippet.
- Audit provenance metadata preservation across stages.
- Detect integrity risks: evidence detachment, span misalignment, clause boundary corruption, chunk misattribution.

### SECTION 3 — Knowledge Graph Construction Integrity
- Analyze transformation of extracted entities/relations into graph-ready objects.
- Audit: node identity strategy, entity deduplication, relation normalization, ontology consistency, graph schema design.
- Detect risks: duplicate nodes, conflicting identities, relation type explosion, schema drift.
- Check entity identifiers for determinism and collision resistance.

### SECTION 4 — Module Dependency and Pipeline Cohesion
- Map dependency structure of NLP extraction modules.
- Identify active modules, referenced but unused, partially integrated.
- Explain dependency chain: module → consumes → produces → next module.
- Evaluate pipeline cohesion vs fragmentation.

### SECTION 5 — Orphan Modules and Dormant Capabilities
- Search codebase for NLP functionality modules not connected to active pipeline.
- List as ORPHAN MODULES with purpose and reason for non-integration.
- Evaluate if integration would improve reliability.

### SECTION 6 — Test Coverage Audit
- Analyze test suite for NLP pipeline adequacy.
- Verify tests for: NER accuracy, entity normalization, relation extraction, semantic chunking boundaries, coreference resolution, graph entity creation.
- Detect missing tests for: edge cases in legal text, nested clauses, ambiguous references, long documents.
- Evaluate if tests verify correctness or only superficial outputs.

### SECTION 7 — Data Integrity Stress Points
- Identify pipeline parts where data corruption may occur.
- Examples: incorrect chunk boundaries, entity duplication, relation misattachment, provenance loss, hash collisions in entity IDs.
- Explain technical conditions for failures.

### SECTION 8 — Hardening Evaluation
- Evaluate hardening against unreliable outputs.
- Check safeguards: confidence thresholds, validation of extracted entities, relation plausibility checks, entity schema validation, graph insertion validation.
- Detect where pipeline may silently produce corrupted/low-confidence data.

### SECTION 9 — Replaceable or Weak Components
- Identify weak components: rule-based entity extraction, naive chunking, string-based entity linking.
- Suggest stronger alternatives where appropriate.

### SECTION 10 — Critical Architectural Risks
- List most serious risks leading to: incorrect legal evidence, broken graph structure, non-traceable legal reasoning.
- Explain root cause and downstream impact.

### SECTION 11 — Refactored Implementations
- Provide improved Python implementations for critical functions:
  - semantic chunking aligned with legal clause boundaries
  - entity normalization with deterministic IDs
  - evidence provenance tracking
  - relation extraction with validation checks
- Focus on reliability, determinism, graph integrity.

    """Generate unique person ID with collision resistance"""

        # Use national ID as primary key when available

    # Combine hash with source document for uniqueness
    name_hash = hashlib.sha256(normalized_name.encode('utf-8')).hexdigest()[:16]
    if source_doc_id:
        doc_hash = hashlib.sha256(source_doc_id.encode('utf-8')).hexdigest()[:8]

    return f"person_{name_hash}"

### 3. Semantic Chunking with Legal Boundaries (`mahoun/pipelines/smart_chunker.py`)
```python
def _simple_chunk(
    self, text: str, doc_id: str, metadata: Optional[Dict[str, Any]] = None
) -> List[Chunk]:
    """
    Fallback simple chunking with legal boundary awareness.
    """
    chunks: List[Any] = []
    start = 0
    chunk_index = 0
    
    # Legal boundary patterns to respect
    legal_boundaries = [
        r'\n\s*ماده\s+\d+',      # Article start
        r'\n\s*بند\s+[الف-ی]',   # Clause start
        r'\n\s*فصل\s+[^\n]',    # Chapter start
        r'\n\s*[\(\[])',         # Parentheses/brackets (often contain lists)
    ]
    
    while start < len(text):
        # Try to find next legal boundary within chunk_size range
        search_end = min(start + self.chunk_size, len(text))
        boundary_pos = search_end
        
        # Look for legal boundaries that would make better chunks
        for pattern in legal_boundaries:
            matches = list(re.finditer(pattern, text[start:search_end]))
            if matches:
                # Take the last boundary that keeps chunk reasonable size
                for match in reversed(matches):
                    potential_end = start + match.start()
                    if potential_end - start >= self.chunk_size // 2:  # At least half target size
                        boundary_pos = potential_end
                        break
                if boundary_pos < search_end:
                    break
        
        end = boundary_pos
        chunk_text = text[start:end].strip()
        
        if chunk_text:
            chunk_metadata = metadata.copy() if metadata else {}
            chunk_metadata["doc_id"] = doc_id
            chunk_metadata["chunk_index"] = chunk_index
            
            # Try to extract structural info
            article_match = re.search(r'ماده\s+(\d+)', chunk_text)
            if article_match:
                chunk_metadata["article_number"] = article_match.group(1)
            
            clause_match = re.search(r'بند\s+([الف-ی]\d+)', chunk_text)
            if clause_match:
                chunk_metadata["clause_number"] = clause_match.group(1)
            
            chunks.append(
                Chunk(
                    text=chunk_text,
                    chunk_id=f"{doc_id}_chunk_{chunk_index}",
                    start=start,
                    end=end,
                    metadata=chunk_metadata,
                )
            )
            chunk_index += 1
        
        # Move start position with overlap, but ensure we don't get stuck
        start = end - self.overlap if end - self.overlap > start else end
        if start >= len(text):
            break
    
    logger.debug(f"Legally-aware chunked document {doc_id} into {len(chunks)} chunks")
    return chunks
```
